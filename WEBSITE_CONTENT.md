Through-line:

> Build the tool, test it with real constraints, explain it clearly, then leave it easier for the next person to use.

### Focus Areas

- Start with the workflow: Hall passes, inventory counts, robot testing, and CRM actions all begin with a real person trying to finish a task.
- Make the system inspectable: Logs, telemetry, permissions, experiment notes, and dashboards make the work easier to debug and easier to trust.
- Ship where people already are: Chrome extensions, mobile scanning, Google Sites, React Admin, and school e-commerce fit the environment instead of forcing a new ritual.
- Turn research into tooling: Quantum experiments, sustainability analysis, and robotics tuning all become clearer when the process is measurable.
- Leave better handoffs: Documentation, reusable subsystems, student mentoring, and cleaner operations help the next person keep moving.

### Working Notes

- Start with the route: Before building, I map who touches the system: student, admin, teammate, mentor, or researcher. That keeps features tied to a real path.
- Test where it breaks: Robots drift, school APIs lag, inventory data gets messy, and quantum hardware is noisy. I try to design around those failure points early.
- Keep tools explainable: I like systems with logs, telemetry, clear permissions, and docs. If someone cannot debug it, they eventually stop trusting it.
- Build for handoff: A project is stronger when another student, team, or staff member can keep using it after the initial push.

### About Traits

- Systems: I think through the data, permissions, UI, and user flow together when I build.
- Research: I like projects where the process is clear, the results can be checked, and the assumptions are written down.
- Follow-through: I care about docs, debugging, handoff, and whether the tool still works after the first version.
- Range: I work across robotics, AI tools, web apps, business operations, and research.
- Team-aware: I try to account for what students, mentors, admins, teammates, and nontechnical users actually need.
- Practical: I prefer tools that make a real task easier, faster, clearer, or easier to repeat.

  - `src/data/experience.ts`
  - `src/data/education.ts`
  - `src/data/leadership.ts`
